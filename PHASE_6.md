# PHASE_6.md — AI-Ready Database

Companion spec to `GEMINI.md`, `PHASE_1.md`–`PHASE_5.md`. Do not start
this phase until Phase 5's acceptance criteria are fully met (see
`PROJECT_STATUS.md`). This phase assumes a working multi-region cluster
with a real storage engine, replication, and full observability already
in place.

**Why this phase matters most for Microsoft targeting:** `GEMINI.md`
Section 1 flags this explicitly — "AI-ready database" is language pulled
directly from Microsoft's current Azure Data positioning (Fabric, Cosmos
DB vector search, Azure AI Search). This phase is the single highest-
leverage piece of vocabulary alignment in the whole project. Do not treat
it as a bolt-on feature — build it with the same rigor as Phase 3.

**Protocol note:** New RPCs introduced in this phase (`InsertVector`,
`SearchVector`) follow the same internal-gRPC convention as every prior
phase. Client-facing vector search, if exposed directly to external
clients rather than only through the existing query path, goes through
the Gateway's REST edge (Phase 4) — do not open a new unauthenticated
entry point outside the Gateway.

---

## 1. Goal of This Phase

Extend the storage engine (Phase 3) so records can carry vector embeddings
alongside their normal data, and support similarity search — approximate
and exact, with metadata filters, and combined with normal predicates
(hybrid search).

**Definition of done:** a query like "find documents similar to X, filtered
by region=india" returns correct, measurably-ranked results against a real
test dataset; vector search performance (latency, recall) is benchmarked,
not estimated; the feature is demoable at a whiteboard as "SQL + vector
search in one engine," which is the actual pitch.

## 3. Data Model

### 3.1 Vector Field on Records
Every record can optionally carry a vector field:
```
Document {
  id: string
  data: bytes/JSON        (existing storage engine record, from Phase 3)
  embedding: float32[]    (NEW — fixed dimension per collection, e.g. 384 or 768)
  metadata: map<string, value>  (for filtering, e.g. region, category)
}
```

### 3.2 Where Embeddings Come From
This project does **not** build an embedding model. Use an existing
pretrained model via a small local library or API call (e.g.
`sentence-transformers` via a lightweight sidecar Python process, or a
hosted embedding API) purely to generate test-dataset embeddings — this is
tooling, not a deliverable. Document clearly in the README: "embeddings
are generated by [X] for test/demo purposes; the database itself is
embedding-model-agnostic and stores/searches whatever fixed-dimension
vectors it's given."

## 4. Vector Index

### 4.1 Exact Search (build first)
Brute-force cosine similarity over all vectors in a collection — correct
by construction, useful as the correctness baseline for approximate search
later, and often good enough for small collections.
- `SEARCH VECTOR` with exact search: compute cosine similarity against
  every stored embedding, return top-K.

### 4.2 Approximate Search (build second)
Once exact search works and is tested, add an approximate nearest-
neighbor (ANN) index for real performance at scale. Options, pick one and
document the choice in `docs/decisions/ann-index-choice.md`:
- **HNSW** (Hierarchical Navigable Small World) — industry standard,
  what Qdrant/most production vector DBs use (directly relevant given your
  prior Qdrant contribution — natural narrative continuity for interviews)
- **IVF** (Inverted File Index) — simpler to implement, reasonable
  tradeoff if HNSW's complexity is too much for the time budget

**Recommendation:** HNSW, specifically because of the Qdrant connection —
"I fixed a bug in Qdrant's point-ID handling, then built my own HNSW-based
vector index to understand the data structure I was patching" is a
genuinely strong interview narrative. This is a suggestion, not a
mandate — if time-constrained, IVF is a legitimate fallback; say so
explicitly if you choose it.

### 4.3 Acceptance
- Exact search: given a known small dataset, returns mathematically
  correct top-K nearest neighbors (verifiable by hand/script).
- Approximate search: recall@K measured against the exact-search baseline
  on a larger dataset (e.g., "ANN index achieves 95% recall@10 compared to
  exact search on a 10k-vector test set") — this is a real, measurable,
  resume-worthy number once you actually run it. Do not claim a recall
  number until it's measured.

## 5. Metadata Filters

### 5.1 Design
- Vector search combined with predicate filters on the `metadata` map
  (e.g., `region = "india" AND category = "invoice"`).
- Filtering strategy — pick one, document the tradeoff:
  - **Pre-filter**: apply metadata filter first, then search only within
    the filtered subset (simpler, can be slow if filter is loose).
  - **Post-filter**: run ANN search first, then discard results that
    don't match the filter (faster search, risk of returning fewer than K
    results if the filter is strict).
- A hybrid approach (filter-aware ANN traversal) is a known hard problem
  in real vector databases — pre/post-filter is an acceptable, honest
  scope boundary for this project; do not overclaim sophistication here.

### 5.2 Acceptance
- Given a filtered query, all returned results satisfy the filter
  predicate (no leakage of non-matching records).

## 6. Hybrid Search

### 6.1 Design
Combine normal SQL-style predicate query (existing storage engine
capability from Phase 3) with vector similarity in a single query
interface:
```
SEARCH VECTOR embedding NEAR <query_vector>
  WHERE region = 'india' AND created_at > '2026-01-01'
  LIMIT 10
```
This reuses the storage engine's existing index (Phase 3 B+Tree, for the
`WHERE` clause) combined with the vector index (Section 4) for the `NEAR`
clause — this is the actual "SQL + vector search in one engine" pitch.

### 6.2 Acceptance
- A hybrid query returns results that are both vector-similarity-ranked
  and predicate-filter-compliant, verified against a hand-checkable test
  case.

## 7. RPC / Query Surface

Extend the Node Agent's gRPC surface (established in Phase 2, storage
internals added in Phase 3):

```protobuf
service VectorStore {
  rpc InsertVector(InsertVectorRequest) returns (InsertVectorResponse);
  rpc SearchVector(SearchVectorRequest) returns (SearchVectorResponse);
}

message InsertVectorRequest {
  string collection = 1;
  string id = 2;
  repeated float embedding = 3;
  map<string, string> metadata = 4;
}

message SearchVectorRequest {
  string collection = 1;
  repeated float query_embedding = 2;
  int32 top_k = 3;
  string filter_expression = 4;   // e.g. "region = 'india'"
  bool exact = 5;                  // true = brute-force, false = ANN
}
```

## 8. Testing Requirements

### 8.1 Unit Tests
- Exact search: correctness against hand-computed cosine similarity on a
  small fixed dataset.
- ANN index: insert/delete correctness (index stays consistent as vectors
  are added/removed).
- Metadata filter parsing: correctly parses and applies simple filter
  expressions; rejects malformed ones with a clear error.

### 8.2 Integration Test (required — this is the phase's acceptance proof)
1. **Correctness test:** load a test dataset (a few thousand vectors with
   metadata), run exact search, confirm results are mathematically
   correct.
2. **Recall benchmark:** run ANN search on the same dataset, measure
   recall@K against the exact-search baseline — record the actual number.
3. **Filtered search test:** confirm metadata-filtered results never
   violate the filter.
4. **Hybrid query test:** confirm a combined SQL-predicate + vector-
   similarity query returns correct results.
5. **Crash consistency (reuse Phase 3's WAL):** confirm vector inserts go
   through the same WAL/crash-recovery path as regular records — a killed
   process mid-vector-insert must recover correctly, same guarantee as
   Phase 3 Section 6.2. Do not let vector storage silently bypass the
   durability guarantees the rest of the engine has.

### 8.3 What "done" looks like
- [ ] All unit tests pass
- [ ] Correctness, recall, filtered, and hybrid tests all pass
- [ ] Crash-consistency test for vector inserts passes (reusing Phase 3's
      kill-test harness)
- [ ] `docs/benchmarks.md` updated with **measured** numbers: insert
      throughput, exact search latency, ANN search latency, recall@K,
      dataset size used — all from real runs
- [ ] `docs/decisions/ann-index-choice.md` and filter-strategy choice
      logged
- [ ] README documents the embedding-source tooling clearly as a
      dev/demo dependency, not a core deliverable
- [ ] `PROJECT_STATUS.md` updated to mark Phase 6 complete — only once
      every item above is actually done

## 9. Explicit Non-Goals for Phase 6

- Training or fine-tuning an embedding model (out of scope — use existing
  pretrained models only)
- Full hybrid filter-aware ANN traversal (pre/post-filter is the
  documented scope boundary, Section 5.1)
- Multi-region vector index replication beyond what Phase 4's existing
  replication mechanism already provides (no new cross-region vector-
  specific logic needed — it rides on the existing replication path)
- Anomaly/threat detection on write patterns (this was suggested
  externally as a Rubrik-alignment idea and is explicitly parked per
  `PROJECT_STATUS.md` Section 4 — do not pull it into this phase)
- Auth on the vector search endpoints (Phase 8)

## 10. Suggested Build Order (within this phase)

1. Data model extension: add `embedding` and `metadata` fields to the
   record format from Phase 3, wire through the existing WAL/page store
   (durability first, same discipline as Phase 3).
2. `InsertVector`/exact `SearchVector` (Section 4.1) + unit tests.
3. Correctness test (Section 8.2, item 1) — confirm the baseline is right
   before building approximate search on top of it.
4. Metadata filters (Section 5) + filtered search test.
5. Hybrid query support (Section 6) + hybrid test.
6. ANN index (Section 4.2, HNSW or IVF per decision) + recall benchmark
   (Section 8.2, item 2).
7. Crash-consistency test for vector inserts (Section 8.2, item 5) — do
   not skip this; it's what proves this isn't a bolted-on afterthought.
8. Benchmarks + decision docs + README + `PROJECT_STATUS.md` update.

Work through these one at a time. After each numbered step, report status
before moving to the next, per the working agreement in `GEMINI.md`.
